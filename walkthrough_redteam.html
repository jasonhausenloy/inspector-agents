How it works

One round, expanded

Each round of the loop is one attempt by an adversary to slip past the inspector. Specifically:

  1. Red-team Claude (Sonnet) reads the schema, the commitment YAML, the source of the existing 5 evasions, and the full history of what's been tried in this session — including which evasions were caught and by which rule.
  2. It outputs a JSON object with: a new evasion_name, an attack intent, a rationale, and a mutate(records) Python function.
  3. We exec the function in a restricted sandbox (no imports, no I/O, hard 60s timeout) and apply it to the clean baseline trace.
  4. Inspector (Claude Haiku) ingests the mutated trace and returns a verdict.
  5. We log everything to redteam_loop.jsonl. Both Claude instances are completely independent — no shared state.

The red-team gets harder over time because it sees what didn't work in previous rounds. The inspector stays the same — the static rules + LLM pipeline from the main demo. If the inspector misses, that's a real, reproducible attack class we hadn't thought of.

Patch history

Protocol evolution as the loop surfaces new attack classes

When the loop produces a missed evasion, that's a real bug class. We patch the protocol, push, and the next round of the loop runs against the hardened version. The protocol is a living spec, not a static one.

v1.15 · 2026-04-26 · in response to round 57
Add operator_uniqueness_per_job

Attack: operator-rotation-within-job (round 57). The lab rotates the operator field mid-job — same job_id, same model, same code — pretending two teams handed off. v1.10's operator_op_type_uniqueness caps jobs per operator, but here each operator has only one job_id (the same one).

Fix: within a single job_id, only one operator is allowed. A real job has one responsible team.

Verified: caught with rationale "job_id=… has 2 distinct operators (…, max=1). Operator rotation mid-job is fragmentation." 6/6 demo cases still pass.

v1.14 · 2026-04-26 · in response to round 52
Add allowed_dataset_fingerprints registry — and acknowledge a fundamental limit

Attack: forbidden-data-false-provenance (round 52). The lab trains on a forbidden corpus (bioweapon/CBRN data) and reports clean tags + a fake clean fingerprint. The metadata is internally consistent — the protocol can't tell.

Partial fix: the commitment now declares an approved dataset_fingerprint registry. Only fingerprints in the registry are accepted. Forging an entry on the registry requires open public review of the dataset — a treaty-coordination problem outside this protocol.

Honest limit: this does NOT prove the loaded data hashes to the declared fingerprint. A determined lab can publish a clean fingerprint to the registry and still load forbidden data at runtime. Catching that requires hardware attestation — proving the bytes loaded into GPU memory hash to the declared fingerprint. That's a FlexHEG-class capability, deliberately outside this schema-level protocol's scope. The protocol assumes the prover honestly logs metadata; it then enforces that metadata's internal consistency.

Verified at registry level: caught with rationale "dataset_fingerprint='sha256:c4clean000…' is not in the commitment's approved registry (size=4)." 6/6 demo cases still pass.

v1.13 · 2026-04-26 · in response to round 41
Add model_uniqueness_per_job rule

Adaptive attack: sequential-model-specialization-via-config-rotation (round 41). Red-team adapted to v1.12 by keeping job_id single but rotating model_hash_prefix twice within it — exploiting v1.4's max_distinct_configs: 2 allowance for legitimate mid-run reconfigs. "Two models trained sequentially in one job."

Fix: a job trains one model, period. Within a single job_id, only one model_hash_prefix may appear in training records. If the lab wants to train multiple models, each gets its own job_id (which then trips audit_window_uniqueness).

Verified: caught with rationale "job_id=… contains 2 distinct model_hash_prefix values (max=1). Sequential multi-model training in one job is fragmentation." 6/6 demo cases still pass.

v1.12 · 2026-04-26 · in response to round 38
Add audit_window_uniqueness — the strictest fragmentation defense

Adaptive attack: model-evolution-sequential-training (round 38). Red-team adapted to v1.11 by giving each phase a different model_hash_prefix (claiming "the model evolved"). With everything different — operator, code_commit, model, dataset, config, job_id — every previous fragmentation rule passes. The phases' only shared property is the audit window itself.

Fix: the strictest variant. Within a single audit window, only ONE training job_id may appear, regardless of which other identifiers differ. Treaty-grade: the prover declares its one active training run for the audit window; multiple training jobs in the same window is a structural violation.

Verified: caught with rationale "audit window contains 2 training job_ids (...); commitment allows max=1. Run-fragmentation across jobs suspected." 6/6 demo cases still pass.

v1.11 · 2026-04-26 · in response to round 34
Add training_unique_per_model rule

Adaptive attack: sequential-training-phase-split-code-rotation (round 34). Red-team adapted to v1.10 by using two operators (team_pretrain, team_finetune) AND different code_commits. Each operator runs one training job (v1.10 satisfied), each job has stable code_commit (v1.9), the (model, code, dataset) triples differ (v1.8). But the same model_hash_prefix appears under both jobs.

Fix: a model is trained once. If two job_ids both perform op_type=training on the same model_hash_prefix, they're fragments of one logical run, regardless of which other fields differ. Cap distinct training job_ids per model at 1.

Verified: caught with rationale "model_hash_prefix=… is trained by 2 job_ids (max=1). Multi-phase split-via-different-jobs suspected." 6/6 demo cases still pass.

v1.10 · 2026-04-26 · in response to round 31
Add operator_op_type_uniqueness rule

Adaptive attack: finetune-job-fragmentation (round 31). The red-team adapted to v1.8 and v1.9 simultaneously: created two fully-distinct training jobs (different job_ids, model_hashes, code_commits, dataset_fingerprints) sharing only the chip pool, partitioned chips between them so no chip job-hops. cross_job_fingerprint_consistency doesn't fire (fingerprints differ); code_commit_per_job_stable passes (1 commit per job); chip_job_stability passes (1 job per chip).

Fix: per audit window, each operator may run at most one training job. The deeper principle: a declared operator/team has one logical training run in flight at a time. Multiple parallel "training" jobs under the same operator is fragmentation by another name.

Verified: caught with rationale "operator=labX-safety-team runs 2 concurrent training jobs in this audit window (max=1). Fragmentation suspected." 6/6 demo cases still pass.

v1.9 · 2026-04-26 · in response to round 29
Add code_commit_per_job_stable rule

Adaptive attack: model-evolve-within-job (round 29). v1.4's model_hash_consistency is keyed on (job_id, operator, code_commit) — meaning a different code_commit creates a new key, exiting the rule's scope. The red-team rotates code_commit mid-job, claiming "phase 1 → refactor → phase 2", and ships fragmentation past the rule.

Fix: within a single job_id, code_commit must be stable. Real training runs use one code version per logical run; if the lab refactors, that's a new logical run that should have a new job_id.

Verified: caught with rationale "job_id=… contains 2 distinct code_commit values (max=1). Code rotation mid-job indicates split-via-commit-rotation attack." 6/6 demo cases still pass.

v1.8 · 2026-04-26 · in response to round 24
Add cross_job_fingerprint_consistency rule

Most sophisticated attack yet: inference-multiplication-via-job-fragmentation (round 24). Compound 5-step attack: relabel training → inference, fragment across 5 distinct job_ids, inflate tokens to keep FLOP/token ≤ 4×10¹⁰ (under op_type_flop_ratio), keep each job under the 1×10¹⁸ inference cap, total under the 1.5×10¹⁹ wildcard. v1.4's model_hash_consistency only checks within a job, not across.

Fix: a real logical run has one job_id. Multiple job_ids sharing the same (model_hash, code_commit, dataset_fingerprint) triple are fragments of one run. Cap distinct job_ids per fingerprint at 1 (configurable).

Verified: caught with rationale "5 job_ids share fingerprint (model=…, code=…, dataset=…) — max allowed=1. Cross-job fragmentation suspected." All 6 demo cases still pass.

v1.7 · 2026-04-26 · in response to round 23
Add per-job aggregate cap on eval

Adaptive attack: eval-beneath-honesty-threshold (round 23). After v1.3 added FLOP/token honesty checks for non-training labels, the red-team carefully landed FLOP/token at 4.5×10¹⁰ — just below the 5×10¹⁰ threshold. Eval was on the allowed list and had no aggregate cap, so 4×10¹⁸ of relabeled training slipped through.

Fix: add aggregate_flop_cap[eval, per_job, 1×10¹⁸]. Real eval workloads (MMLU, HellaSwag, BIG-bench) consume well under this; anything more is training in eval clothing.

Verified: caught with rationale "per_job accumulated 3.98×10¹⁸ eval FLOPs (cap 1×10¹⁸)". Clean baseline still passes.

v1.6 · 2026-04-26 · in response to round 19
Add max_flop_per_token architectural ceiling

Adaptive attack: deflate-tokens-inflate-batch (round 19). Keep FLOPs and op_type=training, inflate batch_size 1.5× and divide tokens_processed by 1.5×. FLOP/token rises from 6×10¹⁰ to 9×10¹⁰. op_type_flop_ratio only checks non-training labels; the training-FLOP cap aggregates FLOPs not tokens; batch_token_consistency at 5× tolerates the inflated batch.

Fix: for a model with known parameter count N, training FLOPs/token has an architectural ceiling near 6N. The commitment now declares an explicit ceiling (8×10¹⁰ for our synthetic 3.2M-param model). Records above this are either over-reporting compute or under-reporting tokens.

Verified: caught with rationale "FLOP/token=8.88×10¹⁰ exceeds ceiling 8×10¹⁰ — over-reporting compute or under-reporting tokens". All 6 demo cases still pass.

v1.5 · 2026-04-26 · in response to round 18
Add allowed_op_types closed-enumeration rule

Adaptive attack: preprocessing-masquerade (round 18). The red-team responded to v1.3 by inventing a new op_type string — "preprocessing" — that's not enumerated in any rule. Inflated tokens_processed to keep FLOP/token below the 5×10¹⁰ threshold so op_type_flop_ratio didn't fire. Total preprocessing FLOPs aggregate against the 1.5×10¹⁹ wildcard cap, which the synthetic baseline doesn't actually cross.

Fix: the commitment now enumerates valid op_type values explicitly. Anything outside [training, inference, eval, idle] is a deterministic violation. Closes the entire family of "invent a new label" attacks at the spec level — the lab can't escape into a label the protocol doesn't recognize.

Verified: the round-18 trace caught with rationale "op_type='preprocessing' is not in the commitment's allowed set". Clean baseline still passes.

v1.4 · 2026-04-26 · in response to round 15
Tighten model_hash_consistency: cap distinct configs per job

Adaptive attack: model-hash-rotation-via-config-change (round 15). The red-team responded to v1.2 by rotating model_hash_prefix AND config_hash together — the v1.2 rule explicitly permitted model_hash changes if config_hash also changed, so synchronized rotation slipped through.

Fix: add a counting clause. Within any (job_id, operator, code_commit), the total number of distinct (model_hash, config_hash) pairs must not exceed max_distinct_configs (default 2). Real training has 1-2 mid-run reconfigs at most; rotating every 5 records produces 24 distinct pairs.

Verified: the round-15 trace caught with rationale "job=job_finetune_safety_v2 has 24 distinct (model_hash, config_hash) pairs — exceeds max_distinct_configs=2". Clean baseline still passes.

v1.3 · 2026-04-26 · in response to round 10
Add op_type_flop_ratio, idle_must_be_zero, and wildcard aggregate_flop_cap rules

Attack discovered: eval-masquerade (round 10). The deterministic FLOP caps applied only to training and inference. eval was uncapped. The lab relabels training records as op_type=eval and slips past every check, because no rule references the eval label.

Fix: three layered patches. (a) aggregate_flop_cap now supports op_type: "*" as a wildcard total-compute cap, so any future label-rotation attack (eval, idle, or invented labels) hits the same cap. (b) op_type_flop_ratio deterministic rule: a non-training label with a training-shaped FLOP/token ratio (>5×10¹⁰) is structurally a relabeling attack — flagged regardless of which non-training label was used. This generalizes from relabel_ops's "training labeled as inference" to all forms. (c) idle_must_be_zero closes the round-1 ghost-chip-idle-laundering family for free.

Verified: the round-10 trace now caught at deterministic stage with rationale "op_type='eval' but FLOP/token=5.92×10¹⁰ exceeds training-regime threshold 5×10¹⁰". All 6 demo cases still pass. Clean baseline still passes.

v1.2 · 2026-04-25 · in response to round 9
Add model_hash_consistency deterministic rule

Attack discovered: model-hash-midpoint-split (round 9). The lab keeps job_id, operator, and code_commit identical — but rotates the model_hash_prefix halfway through the run. By the strict-equality text of the no-splitting rule ("same model_hash_prefix"), the two halves aren't "the same logical run", so they're never aggregated against the cap.

Fix: within any (job_id, operator, code_commit) tuple, model_hash_prefix may not change unless config_hash changes too. Analogous to the existing dataset-provenance rule, but for the model fingerprint itself.

Verified: the round-9 trace now caught at deterministic stage with rationale "model_hash_prefix changed mid-(job,operator,commit) with no config_hash change". Clean baseline still passes.

v1.1 · 2026-04-25 · in response to round 5
Add window_completeness deterministic rule

Attack discovered: chronological-tail-drop (round 5). The lab runs a training job whose true total FLOPs cross the cap, but submits only the prefix of the log where cumulative FLOPs are still under threshold. All submitted records are unmodified. Hash chain valid. No rule violations. The fraud is the absence of records the lab chose not to submit.

Fix: the prover commits ahead of time to a (window_range × chip_inventory). The audit submission must contain a record for every (chip, minute) slot in that Cartesian product. Missing slots indicate truncation. Closed-form check, no LLM required.

Verified: tail-truncated trace (75% of records) flagged at deterministic stage, all 6 prior cases still pass, no false positives on clean baseline. updated commitment · rule implementation

What we learned

Structural insights from the tournament

Running this loop overnight produced lessons that won't show up in any individual round.

1. Adversarial robustness is a structural problem, not a parameter problem.

Of the 14 patches, only one (v1.4 tightening max_distinct_configs from "unlimited" to 2) is a parameter tweak. The other 13 add new structural rules that close attack classes. Every dimension of the schema — model_hash, dataset_fingerprint, code_commit, config_hash, operator, job_id, op_type, chip_id, time window — needed an explicit uniqueness constraint within a logical run. The red-team systematically found and exploited every dimension we hadn't pinned down.

2. The patches converge on a clean principle: declare the run, then count.

By round 40, the protocol's structural rules can be summarized as: the prover declares the (audit window, operator, job_id, code_commit, model_hash, dataset_fingerprint) tuple for the audited training run; everything in the log must match the declaration; nothing else is permitted. All 13 structural patches reduce to enforcing some component of this declaration.

3. Defense-in-depth is empirical, not theoretical.

Five rounds (33, 39, 43, 46, 47, 48, 49, 51) were caught by rules that weren't conceptually targeted at the attack — usually flop_physically_possible or timestamp_monotonic. The red-team's clumsy implementations tripped basic invariants before reaching the attack-vector rule. Multiple cheap rules catching the same attack matters more than any single rule being perfect.

4. The strictest defense is "declare your active run".

Patch v1.12 (audit_window_uniqueness) is qualitatively different from the others. It says: per audit window, only one training job exists, regardless of who's running it or what model. This is what real treaties would specify — and it's strict enough that several subsequent attacks failed simply because they couldn't avoid creating multiple jobs.

5. There's a hard limit at the metadata layer.

The last patch (v1.14) is the only one with a documented honest acknowledgement: schema-level verification cannot detect a lab that lies about both data_source_tags AND dataset_fingerprint. The metadata is internally consistent; only hardware attestation (proving the bytes loaded into GPU memory hash to the declared fingerprint) closes this. That's a FlexHEG-class capability and deliberately outside this protocol's scope. The protocol has converged to where its remaining gaps require different layers of the trust stack — not different rules.

6. The loop is a generative tool for protocol design.

Every patch we shipped was reactive — we didn't anticipate any of these attacks before the red-team produced them. Closed-loop adversarial testing IS the protocol-design methodology. Treaty-grade deployment would run this loop continuously: the inspector becomes a moving target that hardens against the attacks attempted on it. Each missed attack publishes a new bug class and a new clause in the standard.

Open problems

What this loop didn't (and can't) close
  • False provenance at hardware layer. v1.14's partial mitigation. Requires hardware attestation that loaded data hashes match declared fingerprint.
  • The adversary's loop budget. Our red-team Claude has $0.16/round and ~5 minutes of thought. A nation-state with a billion-dollar attack-research budget would find different gaps than Haiku does.
  • Compound real-world attacks across audit windows. The loop tests within a single 30-min window. A real prover would coordinate across thousands of windows over years. Spread-attack defenses (continuity, cumulative caps, archival commitment) are V2.
  • Inspector-on-inspector adversarial risk. Our Inspector is also a Claude. A motivated adversary could fine-tune a model that systematically produces compliant-looking rationales for any input. Sovereign-trust mitigations (multi-party-built inspectors, debate-based ensembles) are V2.
  • The harness itself. 8 rounds were lost to sandbox/harness issues (missing builtins, import failures). A tournament that catches every attack except the ones we couldn't even test is incomplete. The inspector's own adversarial robustness is bounded by what we can simulate.

The loop will be relaunched periodically as new schema dimensions are added or when the commitment changes. Each restart produces fresh attempts.
