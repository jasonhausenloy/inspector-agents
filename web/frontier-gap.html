The Verifier Challenge · Honest limits

What this can't show yet.

A single-chip laptop run is the wrong scale to verify a frontier model. The architecture is the same; the gap is real and the Verifier Challenge funds closing it.

What we capture honestly

Per training step on a MacBook M3:

  • Loss, gradient norms (total + per parameter group), optimizer state norms (AdamW first and second moments), and per-50-step weight norms.
  • Latency decomposition: data-load, forward, backward, optimizer step.
  • System telemetry: process RSS, CPU load, MPS-allocated memory, system load average.
  • Top-5 largest tensors with min/max/mean/std distribution snapshots, every 50 steps.
  • Token IDs sampled every 100 steps so the verifier can prove the trace is using the declared dataset.
  • SHA-256 hash chain across all records.

Per generated inference token:

  • Sampled token ID and string.
  • Full top-10 logits with probabilities, plus entropy and top-1 probability across the entire vocab.
  • KL divergence from the previous step's distribution.
  • Per-layer per-head attention summary: entropy and the top-3 attended key positions for the last query position.
  • KV-cache size equivalent, context length, latency breakdown.

What's missing — and what would close the gap

Per-chip telemetry that NVIDIA exposes and Apple Silicon doesn't

  • DCGM-style tensor-core utilization. NVIDIA chips expose SM_ACTIVE, PIPE_TENSOR_ACTIVE, and MEM_COPY_UTIL as first-class counters. Apple Silicon does not. A frontier verifier would use these to detect shadow compute (chip running a parallel job during another job's idle slots).
  • Per-GPU power draw. Frontier datacenters log W and J directly. We don't.
  • Thermal headroom and clock-frequency throttling. Used to detect off-spec workloads.
  • ECC error counters. Used to detect tampering with hardware state.

Multi-host coordination signals that don't exist on a single chip

  • All-reduce latency for gradient sync. Frontier training spreads each step across hundreds of GPUs and synchronizes gradients. The latency profile of those all-reduces is one of the strongest signals that a workload is actually training.
  • Pipeline-parallel bubble time. Indicates the model size is genuinely larger than one chip can hold.
  • Per-rank loss and gradient norms across hundreds of ranks. Stragglers, divergent ranks, restarts.
  • Network fabric counters. InfiniBand RDMA, NVLink utilization, link errors.

Content detail we sample, not stream

  • We capture top-5 tensor stats every 50 steps. Frontier verification wants every tensor every step.
  • We capture activation statistics only for the last query position in inference. Frontier wants per-layer activation distributions for all positions.
  • We capture gradient norms only — not gradient histograms. Histograms catch attacks where a lab forges grad norms but can't fake the distribution shape.

Cryptographic attestation

  • Our hash chain is content-only — it proves the records weren't tampered with after writing, but not that they faithfully reflect the workload that ran. Frontier verification needs hardware attestation (signed by the GPU) that the loaded data hashes to the declared dataset_fingerprint and the loaded weights hash to the declared model_hash_prefix.
  • That requires either NVIDIA Confidential Computing-style attested execution, or a TPM-style trusted boot path, neither of which we exercise here.

What this means for the Challenge

The single-chip MVP proves the architecture works at one chip: a real GPT, a schema-compliant log, a Claude verifier that catches every adversarial mutation we hand-built. That is the proof of concept. The Verifier Challenge funds the work that closes the items above — which is most of what makes large-scale verification hard.

The honest framing. On a per-chip basis, our trace captures roughly 70-80% of what an auditor would want. On a multi-host basis, perhaps 10%. On per-tensor / per-layer content detail, perhaps 30%. Each gap is a Challenge research direction.
