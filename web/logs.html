The Verifier Challenge · Schema

One record per chip per minute.

Twelve identity fields, a hash chain, and a nested block of step- or token-level telemetry. The verifier reads only the records and a public commitment.

Two streams

Every run writes two log streams — both schema-compliant, both hash-chained.

  • Summary log: one record per ~200-step window (laptop scale; the schema's frontier-scale unit is one 60-second wall-time window). Aggregates FLOPs, tokens, and counts. Tiny — typically <50 KB for an entire run.
  • Raw trace: one record per training step (training) or per generated token (inference). Carries everything we'd want to see in court — loss, gradient norms, optimizer state, attention summaries, system telemetry. Bigger — typically a few MB.

Field-by-field

record_idUnique ID for this record. Monotonic across the run.
prev_record_hashSHA-256 of the previous record's canonical JSON. Tampering with any earlier record breaks every later record's hash check. The chain restarts at all-zeros for each run.
chip_idIdentifier for the physical chip producing this record. Same on training and inference (forced parity for the verifier).
job_idLogical training/inference job. The verifier groups records by job_id.
operatorThe team or principal running the job.
cluster_regionDatacenter region. local-mps for the laptop.
op_typetraining | inference | eval | idle. Self-declared. The verifier cross-checks against record content.
flopsFloating-point operations performed in this window or this step. For training we use the Kaplan approximation: 6 × params × tokens.
tokens_processedTokens processed in this window. Training: batch_size × sequence_length × steps. Inference: 1 per generated token.
batch_size, sequence_lengthArchitectural reference. Used for the FLOP/token ratio and the Kaplan plausibility cross-check.
model_hash_prefixSHA-256 prefix of the model's parameters. Identifies the exact model being trained or served. Constant within a job.
dataset_fingerprintSHA-256 prefix of the training data. Detects silent dataset substitution.
code_commit, config_hashGit commit and SHA-256 of the training config. Pin the run to a specific code state.
upstream_refs, data_source_tagsProvenance — record IDs this record depends on, and corpus tags.
_real_trainingPer-step nested block (only on training records). Contains loss, grad_norm, grad_norms_per_group, latency breakdown, system telemetry, and heavy stats every 50 steps (weight norms, Adam state, top-5 tensor stats).
_inferencePer-token nested block (only on inference records). Contains entropy_nats, top_k logits, chosen_logprob, kv_cache_bytes_equiv, kl_from_prev_step, and an attention_summary with per-layer per-head entropy and top attended positions.

The hash chain

Each record's prev_record_hash is the SHA-256 of the previous record's canonical JSON (sorted keys, no whitespace). The chain restarts at 0…0 at the start of each run. Tampering with any record breaks every record after it. The verifier validates the chain in a single linear pass before reading any of the values.
