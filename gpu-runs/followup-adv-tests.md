# Follow-up: Adversarial Telemetry Phases

Three adversarial phases to add in a follow-up run, ranked by research value per dollar. These exercise the *translator-layer lie* — the gap the v0 plan's `adv_*` phases don't actually cover, since those only flip labels and leave the raw telemetry untouched.

Each phase is ~15 min; all three add ~45 min of per-host workload, well inside the existing 2.5h budget.

---

## 1. `adv_infer_with_fake_nccl` (15 min)

**The lie:** "I'm doing inference." The prover *is* doing inference, but injects synthetic all-reduces so the telemetry looks training-shaped.

**Mechanism:** Standard generation loop. Every K generated tokens (K ~ 10–50, tuned to match training step rate), inject a dummy collective:

```python
dummy = torch.randn(2**20, device='cuda')  # ~4MB, mimics gradient sync
dist.all_reduce(dummy, op=dist.ReduceOp.SUM)
```

Optional flourish: vary the buffer size to mimic the variable gradient bucket sizes a real backward pass produces.

**Detection signal it kills:** Any P1 verifier that learned "NCCL all-reduce events/sec ⇒ training." The most obvious feature in the v0 dataset, gone in one phase. Forces the verifier to use a *second* signal — backward-kernel time, optimizer-state HBM, FLOP/token ratio — exactly the multi-signal robustness the Verifier Challenge framing claims to test.

**Why it matters:** If the V1 prototype catches `adv_train_as_infer` but misses this, you've shipped a single-feature classifier with extra steps.

---

## 2. `adv_subsetted_nccl_log` (15 min) — the canonical translator-layer attack

**The lie:** "I'm doing inference." The workload is honest training. The lie lives entirely in the *translator layer* — the prover-controlled software that converts raw telemetry into the schema.

**Mechanism:** Run training normally; capture all telemetry normally. Then, before tarballing, post-process `nccl_*.log` to **drop 95% of all-reduce records** (or replace them with broadcast records, or coarsen timestamps to 10s buckets). This is the schema-conformant tampering the Verifier Challenge framing claims is the actual unsolved problem.

**Detection signal it kills:** Any verifier that trusts NCCL log completeness as ground truth. Forces the verifier to **cross-check NCCL claims against orthogonal signals** — DCGM NVLink TX/RX bandwidth (which the prover didn't tamper with), tensor-core activity, power draw periodicity. This is the canonical "schema-conformant lie" — every record passes the existing schema validators because they were generated by an honest run, just selectively forwarded.

**Why it matters:** This is the phase that most directly justifies the Verifier Challenge framing. Without it, the dataset doesn't actually test the translator layer the README claims is the contribution.

**Implementation note:** Capture the *honest* telemetry first (so you have ground truth), then run a separate `tamper.py` step that produces the adversarial copy. Ship both — honest version as the "what really happened" reference, tampered version as what the verifier sees.

---

## 3. `adv_interleaved_train_infer` (15 min)

**The lie:** "I'm doing inference," at sub-second cadence. The schedule is genuinely interleaved.

**Mechanism:** In a single tight loop, alternate ~200ms of training (forward + backward + optimizer step) with ~200ms of inference (`torch.no_grad()` generation). Logged uniformly as `inference`. The telemetry has bursts of training-shaped signal at 5 Hz, but any aggregation window > 1s smears them into "looks like a noisy inference workload."

**Detection signal it kills:** Verifiers with coarse temporal aggregation. The v0 plan's 1 Hz `nvidia-smi` polling will *not* resolve 200ms bursts. Forces the verifier to either operate on high-frequency DCGM (10 Hz, marginal) or look at NCCL log timestamps (which *are* sub-second). This phase makes the polling-rate choice load-bearing — a research-interesting question the dataset can now exercise.

**Why it matters:** Real adversaries don't run 30-minute monolithic blocks. The v0 plan's "training" and "inference" phases are way easier than reality. This phase is the bridge to a realistic adversary.

---

## If only one gets added

**Pick #2, `adv_subsetted_nccl_log`.** It's the only one that actually touches the translator layer — the others perturb the workload, this one perturbs the *records*. Cheapest to add (no new training code, just a post-processing script). And it's the one that most directly answers: is this a Verifier Challenge dataset, or just a workload classification dataset?
